> **Purpose**: This document describes the entire krecon codebase in enough detail that an LLM can answer questions, generate code, debug issues, and extend features without needing the raw source files.

## 1. Project Overview

**KRecon** is a **Creator-Brand Marketplace** platform built with **Python / FastAPI**. It connects content *Creators* (influencers) with *Brands* (companies), allowing brands to list products and use an **AI-powered matching engine** (sentence-transformers) to find the best creators for their campaigns.

### Tech Stack

| Layer | Technology |
|---|---|
| **Framework** | FastAPI |
| **ORM** | SQLAlchemy (declarative base) |
| **Database** | SQLite (`krecon.db`) |
| **Auth** | JWT (python-jose) + bcrypt password hashing (passlib) |
| **AI Matching** | `sentence-transformers` (model: `all-MiniLM-L6-v2`) + numpy cosine similarity |
| **Environment** | python-dotenv for `.env` loading |

### Key Dependencies (from requirements.txt)

- `fastapi`, `uvicorn` ‚Äî web framework & ASGI server
- `sqlalchemy` ‚Äî ORM
- `python-jose[cryptography]` ‚Äî JWT encoding/decoding
- `passlib[bcrypt]` ‚Äî password hashing
- `sentence-transformers` ‚Äî semantic text embeddings for AI matching
- `numpy` ‚Äî vector operations
- `python-dotenv` ‚Äî env var loading

## 4. Database Setup ‚Äî `backend/database.py`

Sets up the SQLAlchemy engine, session factory, and declarative base.

- **Engine**: `create_engine("sqlite:///./krecon.db", connect_args={"check_same_thread": False})`
  - `check_same_thread=False` is required for SQLite with FastAPI's async workers.
- **SessionLocal**: `sessionmaker(autocommit=False, autoflush=False, bind=engine)`
- **Base**: `declarative_base()` ‚Äî all models inherit from this.

Every route file defines its own `get_db()` dependency that yields a `SessionLocal()` session.

## 7. Authentication System ‚Äî `backend/routes/auth.py`

### Configuration (hardcoded)
- `JWT_SECRET = "supersecretkey"`
- `JWT_ALGORITHM = "HS256"`
- `ACCESS_TOKEN_EXPIRE_MINUTES = 30`
- OAuth2 scheme: `OAuth2PasswordBearer(tokenUrl="/auth/login")`
- Password hashing: `CryptContext(schemes=["bcrypt"], deprecated="auto")`

### Endpoints

#### `POST /auth/register`
Registers a new user.

**Query Parameters**: `name`, `email`, `password`, `role`

**Logic**:
1. Check if email already exists ‚Üí 400 error if yes
2. Hash password with bcrypt
3. Create `User` record with given `role` (should be `"creator"` or `"brand"`)
4. Return `{"message": "User registered successfully"}`

#### `POST /auth/login`
Authenticates a user and returns a JWT.

**Query Parameters**: `email`, `password`

**Logic**:
1. Find user by email
2. Verify password with bcrypt ‚Üí 400 error if invalid
3. Generate JWT with `sub` = user ID, `exp` = now + 30 minutes
4. Return `{"access_token": "<jwt_token>"}`

#### `GET /auth/me`
Returns the current authenticated user's info.

**Headers**: `Authorization: Bearer <token>`

**Response**: `{"id": ..., "email": ..., "role": ...}`

### Helper Functions

- **`create_access_token(data: dict)`**: Creates a JWT with expiration.
- **`get_current_user(token, db)`**: Dependency that decodes JWT, extracts user ID from `sub` claim, looks up user in DB. Used by all protected endpoints.
- **`get_db()`**: Database session dependency.

### ‚öÝÔ∏è Known Bug
Line 24: `db.close` should be `db.close()` (missing parentheses ‚Äî session never actually closes).

Prefix: `/creator` | Tags: `["creator"]`

### Endpoints

#### `POST /creator/profile`
Creates a creator profile for the authenticated user.

**Query Parameters**: `bio`, `niche`, `followrs` (note: typo in parameter name)

**Logic**:
1. Verify `current_user.role == "creator"` ‚Üí 403 if not
2. Check if profile already exists ‚Üí 400 if yes
3. Create `Creator` record
4. Return `{"message": "Creator Profile created Sucessfully"}`

**‚öÝÔ∏è Known Bug**: Parameter is named `followrs` (typo), but the Creator model column is `followers`. This will cause a runtime error (`Creator(followrs=...)` is invalid).

Searches/filters creators with pagination.

**Query Parameters** (all optional):
- `niche` ‚Äî filter by exact niche match
- `min_followers` ‚Äî minimum follower count
- `max_followers` ‚Äî maximum follower count
- `limit` (default 10) ‚Äî page size
- `offset` (default 0) ‚Äî pagination offset
- `sort_by_followers` (default False) ‚Äî sort descending by followers

**Response**: Array of creator objects with nested user info:
```json
[{
    "creator_id": 1,
    "bio": "...",
    "niche": "tech",
    "followers": 50000,
    "user": { "id": 1, "name": "...", "email": "..." }
}]
```

#### `GET /creator/{creator_id}`
Gets a single creator's profile by ID.

**Path Parameter**: `creator_id` (int)

## 11. AI Matching System ‚Äî `backend/routes/ai.py` + `backend/ai/matcher.py`

### Route: `POST /ai/match-creators`

Prefix: `/ai` | Tags: `["AI Matching"]`

Requires authentication.

**Query Parameter**: `prompt` (str) ‚Äî a natural language description of the type of creator a brand is looking for (e.g. "fitness influencer with large following")

**Logic**:
1. Fetch all creators from DB
2. Pass `prompt` and `creators` list to `match_creators()`
3. Return top 5 matched creators

**Response**:
```json
[{
    "id": 1,
    "niche": "fitness",
    "followers": 100000,
    "bio": "Fitness guru..."
}]
```

### Matching Algorithm: `backend/ai/matcher.py`

Uses the `all-MiniLM-L6-v2` sentence-transformer model.

**`match_creators(prompt, creators)`**:
1. Encode the `prompt` string into a 384-dim embedding
2. For each creator, build a text string: `"{niche} {bio} {followers}"`
3. Encode each creator's text into an embedding
4. Compute cosine similarity between prompt embedding and each creator embedding
5. Sort by similarity score (descending)
6. Return the top 5 creators

**`cosine_similarity(a, b)`**: Manual implementation using numpy: `dot(a,b) / (norm(a) * norm(b))`

## 13. Authentication Flow

```
1. User registers via POST /auth/register (name, email, password, role)
2. Password is hashed with bcrypt and stored
3. User logs in via POST /auth/login (email, password)
4. Server verifies password, generates JWT with { sub: user_id, exp: +30min }
5. Client includes JWT in Authorization header: "Bearer <token>"
6. Protected endpoints use get_current_user dependency to:
   a. Decode JWT
   b. Extract user_id from "sub" claim
   c. Query User from DB
   d. Return User object (or 401 if invalid)
```

## 18. Architecture Patterns Used

1. **Dependency Injection**: FastAPI's `Depends()` for DB sessions and auth
2. **Repository Pattern (implicit)**: Direct SQLAlchemy queries in route handlers (no separate repository layer)
3. **Role-Based Access Control**: Checked via `current_user.role` in each endpoint
4. **Semantic Search**: Sentence-transformer embeddings for creator-brand matching
5. **Auto-migration**: `Base.metadata.create_all()` for auto-creating tables on startup (no Alembic migrations)
